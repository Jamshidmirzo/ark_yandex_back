# Аудит прав доступа car-orders — находки и тестовое покрытие

Сквозной прогон permission-тестов по API `car_orders` (слой enforcement, на который опираются и
Flutter-приложение, и React-веб). Цель: прогнать **каждый** эндпоинт по **каждой** роли при
`REQUIRE_OVERLAY_AUTH=True` и поймать, где бэкенд ломается, течёт или расходится с тем, что
предполагают клиенты.

## Что добавлено

| Файл | Покрытие |
|------|----------|
| `tests/conftest.py` | `auth_client` (фабрика клиента через demo-bridge), `make_user` (реальный Django-пользователь + AccessGroup), наборы ролей (`DRIVER`, `DISPATCHER`, `CREATOR`, `GARAGE`, `HYBRID`, …) |
| `tests/test_permissions_overlay.py` | матрица «каждый overlay-маршрут × 5 ролей» + анти-спуфинг (`assignee_driver_id`) + публичные `estimate`/`live-location GET` + неавторизованный = 401 |
| `tests/test_permissions_native.py` | матрица «каждое действие нативного viewset × 8 ролей» (`_action_permissions` + инлайн-`user_has_permission` + карта garage) + HTTP-проверка `cancel`/`reject` |
| `tests/test_permissions_ownership.py` | ось **владения** (object-level / IDOR): водитель не может действовать на чужом заказе; `trip_state` нельзя протолкнуть в обход машины состояний |
| `tests/test_views_errors.py` | отрефакторен на общий `auth_client` (поведение не меняется) |

Запуск: `.venv/bin/pytest car_orders/tests/ -q` → **392 passed**.

Уровни прав, которые фиксируют тесты:

- **публичные** — `estimate`, `live-location` GET (открыты даже при enforcement, по дизайну).
- **любой авторизованный** — чтения, данные, ограниченные самим вызывающим, общие командные инструменты.
- **диспетчер** (`car_order:approve`) — дашборды + привилегированные записи.
- **гейты по нативным codename** — `create`→`car_order:create`, `approve/reassign`→`car_order:approve`,
  `claim/release`→`driver:accept_order`, `start/complete`→`driver:trip_control`,
  garage CRUD→`garage:*`, `drivers/`→`driver:list`, `make/remove-driver`→`driver:assign_to_user`,
  `vehicle-reports`→`vehicle_report:create`. Проверено, что ARK-иерархия (`administrator` ⊇ всё,
  `X_all` ⊇ `X`) удовлетворяет гейтам.

## §B — ИСПРАВЛЕНО: диспетчерские дашборды читались любым авторизованным пользователем

`FleetLiveView`, `DriverPositionsView`, `DriverShiftsView` были закрыты только
`OverlayAuthenticated`, поэтому **любой** авторизованный токен (клиент, водитель) мог прочитать
всю диспетчерскую доску — каждый активный заказ + живую позицию + флаги риска, местоположение
каждого водителя, полный реестр находящихся на смене. В докстрингах сами эндпоинты названы
«диспетчерскими», и в **обоих** клиентах их вызывают только диспетчерские экраны (веб
`FleetLivePage`, мобайл `lib/features/dispatcher/…`).

**Исправление** (`car_orders/views.py`): три view теперь используют `permission_classes =
[OverlayDispatcher]` (`car_order:approve`). Клиенты не ломаются — водители/клиенты их не вызывают.
Зафиксировано строками `DISP` overlay-матрицы.

## §A — ИСПРАВЛЕНО: overlay-эндпоинты *мутации* принимали любого авторизованного актёра

Эти эндпоинты МУТИРУЮТ overlay-состояние claim / trip / shift, но требовали лишь `OverlayAuthenticated`:

| Эндпоинт | Было | Нативный близнец требует | Риск |
|----------|------|--------------------------|------|
| `POST …/{id}/overlay-claim/` | любой авториз. | `claim` → `driver:accept_order` | токен уровня клиента может забрать заказ (на себя) |
| `POST …/{id}/overlay-release/` | любой авториз. | `release` → `driver:accept_order` | любой авториз. может снести claim водителя |
| `POST …/{id}/extend/` | любой авториз. | докстринг: «водитель или диспетчер» | любой авториз. может продлить окно любого заказа |
| `POST …/drivers/me/location/` | любой авториз. | `my_location` → `driver:accept_order` | не-водитель может вписать `DriverPosition` |
| `PATCH/DELETE …/drivers/me/shift/` | любой авториз. | `my_shift` → `driver:accept_order` | не-водитель может зарегистрировать фантомную смену |

Идентичность была защищена и раньше (`assignee_driver_id`/`acting_driver_id` берут актёра из
токена; подменённый в теле `driver_id` игнорируется для не-диспетчеров — зафиксировано
анти-спуфинг-тестами). Брешь была в том, что **роль** не проверялась, и overlay-слой был слабее
своего нативного близнеца.

**Исправление** (`car_orders/permissions.py`, `car_orders/views.py`): новый permission-класс
`OverlayDriverOrDispatcher` (`driver:accept_order` ∨ `car_order:approve`, с ARK-иерархией +
суперпользователь) теперь закрывает `overlay-claim`, `overlay-release`, `extend`,
`drivers/me/location` (POST) и `drivers/me/shift` (PATCH/DELETE). Токен уровня клиента получает
отказ; реальные водители/диспетчеры не затронуты. Единственный AUDIT-тест, который авторизовывал
пользователя *без прав* для overlay-claim, теперь авторизует водителя с `driver:accept_order`
(реалистичный случай — его анти-спуфинг-проверка не изменилась). Зафиксировано строками `DOD`
overlay-матрицы.

## §C — ИСПРАВЛЕНО (на стороне веба): кнопка «Завершить» vs нативный гейт `complete`/`start`

Нативные `start`/`complete` требуют `driver:trip_control` (правильно — управление поездкой это
способность водителя). Веб показывал кнопку завершения *любому* назначенному водителю **или**
диспетчеру и на не-overlay-пути вызывал нативный `complete`
([`OrderWorkflowActions.tsx:232`](../../ark_yandex_front/src/components/OrderWorkflowActions.tsx#L232)).
Поэтому диспетчер (`car_order:approve`, без `driver:trip_control`) видел кнопку, но получал **403** —
видимое действие, которое падает.

Нативный гейт — правильный контракт и **зафиксирован** (строки `complete`/`start` нативной матрицы:
диспетчеру отказано). Рассинхрон был на стороне веба. **Исправление (веб):** вызывать нативный
`complete()` только когда у пользователя есть `driver:trip_control`; диспетчер закрывает заказ
через `setTripState("completed")` (что overlay уже разрешает диспетчеру). Логика вынесена в
`shouldCallNativeComplete` (`src/components/orderWorkflowGates.ts`) и покрыта тестом.

## §D — ЗАМЕЧАНИЕ (вне локальной зоны): «create-direct» не выполняется нативным viewset

Веб/мобайл предполагают, что `car_order:approve` позволяет создать заказ сразу в `awaiting_driver`
(минуя согласование). Нативный `CarOrderViewSet.create` всегда форсит `status=DRAFT`
([`views.py:1047`](views.py#L1047)). В проде этот путь проксируется на вышестоящий demo-бэкенд,
который и владеет поведением create-direct — поэтому локально оно **не** энфорсится и не тестируется
здесь. Отмечено для осведомлённости; изменений нет.

## §E — ИСПРАВЛЕНО (раунд 2, найдено состязательной проверкой): дыры в проверке владения

Матрица ролей доказывала, *кто может вызвать* эндпоинт, но второй, состязательный проход нашёл
ортогональную ось, которую она не покрывала: пользователь, прошедший гейт по роли, всё ещё мог
действовать на заказе **другого водителя**, и существовал обход машины состояний поездки. Всё
подтверждено по исходникам + ранранным пробником, затем исправлено и зафиксировано в
`tests/test_permissions_ownership.py`:

1. **`trip_state` можно было записать через обычный upsert `/meta/` (HIGH).** `OrderMetaSerializer`
   отдавал `trip_state` как записываемое поле, а `OrderMetaView.post` срезал только поля *назначения* —
   поэтому любой авторизованный мог `POST {"trip_state":"completed"}` на `/meta/` и принудительно
   завершить ЛЮБОЙ заказ, минуя всю машину состояний `TripStateView` (переходы, геозону прибытия,
   проверку назначенного водителя, сайд-эффекты). **Исправление:** `trip_state` теперь `read_only` в
   сериализаторе (ни один клиент не выставлял его через `saveMeta`); он может двигаться только через
   `TripStateView`.
2. **У `overlay-release` / `overlay-extend` не было проверки владения (HIGH).** После §A они требуют
   *роль* водителя/диспетчера, но `services.overlay.release/extend` работали чисто по `order_id` —
   водитель мог снять/продлить заказ **коллеги** (их нативные близнецы `services.orders.release/extend`
   проверку владения *делают*). **Исправление:** добавлен `_ensure_actor_owns` (водитель владеет
   заказом, либо диспетчер, либо доверенный серверный вызов с `actor_driver_id=None`); view теперь
   прокидывают актёра из токена.
3. **`trip-state` на заказе без водителя + открытый гейт (MEDIUM).** `TripStateView` был
   `OverlayAuthenticated`, а `trip_state.validate` пропускал проверку владельца при `driver_id is None`
   — поэтому любой авторизованный мог двигать машину состояний неназначенного заказа. **Исправление:**
   view теперь `OverlayDriverOrDispatcher`, а `validate` отказывает не-диспетчеру-водителю на заказе без
   водителя.

## Открытые пункты (помечены, ждут вашего решения — автоматически не менялись)

Их проверочный проход оценил как *требующие решения* (продуктовая политика, а не явный баг):

- **Чтения overlay-метаданных не ограничены по владельцу.** `GET /{id}/meta/`, `meta-batch`,
  `claim-check(-batch)` закрыты `OverlayAuthenticated` и принимают произвольные `order_id` — любой
  авторизованный токен может пакетно прочитать координаты / плановое окно / назначение водителя любого
  заказа. Решение: достаточно ли чувствительны overlay-метаданные, чтобы ограничить их вызывающим
  (своими / только диспетчер)?
- **POST на `/meta/` всё ещё позволяет не-диспетчеру перезаписать поля окна/координат**
  (`planned_datetime`, `estimated_duration`, координаты `origin/address`, `is_urgent`, `has_return`, …)
  на любом заказе. Поля *назначения* защищены (выпущено); эти поля расписания — нет. Опциональное
  усиление.
- **Веб-teardown при примирении (reconcile).** `CarOrderDetailPage` авто-вызывает `overlayRelease` для
  смотрящего на терминальный заказ; после §A смотрящий уровня клиента теперь получает (проглоченный,
  `.catch`) 403. Видимого для пользователя слома нет — серверный хук `reject_overlay` делает реальный
  teardown — но у *отменённого* заказа эквивалентного серверного хука нет, поэтому чистое решение это
  серверный хук на cancel (по аналогии с reject), чтобы терминальный teardown никогда не зависел от
  вызова клиента без прав.

## Итог

- **§B исправлено** — диспетчерские дашборды (fleet/positions/shifts) больше не утекают любому токену.
- **§A исправлено** — overlay-эндпоинты мутации теперь требуют водителя-или-диспетчера (паритет с нативными гейтами).
- **§C исправлено на стороне веба** — веб вызывает нативный `complete()` только при наличии `driver:trip_control`; диспетчер закрывает через `setTripState`. Нативный гейт остаётся корректным.
- **§D** — create-direct это поведение вышестоящего demo; отмечено, локальных изменений нет.
- **§E исправлено** — три дыры уровня владения / обхода машины состояний (одна HIGH-обход trip_state, две HIGH-IDOR между водителями, одна MEDIUM-гейт для заказа без водителя), найденные состязательным проходом, который сама матрица ролей пропустила.
- Нативные гейты по codename и анти-спуфинг-логика идентичности **корректны** и теперь покрыты регрессией (**392 backend-теста зелёные**), плюс мобайл 18 + веб 12.
- **Открытые пункты** выше — продуктовые решения, оставленные на ваше усмотрение.
