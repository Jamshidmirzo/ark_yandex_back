## Краткое резюме

Самое важное предложение: **ключевое бизнес-правило — «1 водитель =
1 активный заказ» — не является concurrency-safe, и на дефолтной базе данных блокировки
строк, призванные его защитить, молча отключены.** Наложенное поверх этого
положение «аутентификация выключена по умолчанию», при котором неаутентифицированные вызывающие стороны могут писать позиции
заказов, приводит к тому, что находки с наивысшей серьёзностью группируются вокруг *конкурентности* и
*авторизации*, а не вокруг (хорошо покрытой) бизнес-логики.

| Серьёзность | Кол-во | ID |
|----------|-------|-----|
| Critical | 3 | C1, C2, C3 |
| High | 5 | H1, H2, H3, H4, H5 |
| Medium | 6 | M1–M6 |
| Low | 3 | L1–L3 |

## Предпосылки окружения (этими предпосылками обрамлена каждая находка) ✔

- **БД по умолчанию SQLite** ([config/settings.py:117-122](../../config/settings.py#L117-L122)); в `pyproject.toml` нет `DATABASE_URL`. На SQLite `select_for_update()` в Django — это **молчаливый no-op** (backend выставляет `has_select_for_update = False`). Это справедливо и для 100% набора тестов.
- **`REQUIRE_OVERLAY_AUTH` по умолчанию `False`** ([config/settings.py:197](../../config/settings.py#L197)) → оверлейные эндпойнты работают полностью открыто, если переменная окружения не задана.
- **Нет `ATOMIC_REQUESTS`** — транзакции существуют только там, где `atomic()` написан явно.
- Channel layer по умолчанию in-memory (попроцессный, теряется при перезапуске); Redis только если задан `REDIS_URL`.

## Граница доверия (кто что поставляет)

```
 browser / driver app ──▶ /api/v1/car-orders/*  ──┬─▶ local overlay views (OrderMeta, trip-state, claim, live-loc)
                          (DemoTokenAuthentication │   identity from token IF REQUIRE_OVERLAY_AUTH else from body/query
                           or AllowAny)            └─▶ gateway ──▶ upstream demo backend (login + base order data)
```
Поставляемый в теле `driver_id` и путевой `order_id` — это недоверенные входные данные;
identity выводится из demo-токена **только когда `REQUIRE_OVERLAY_AUTH=True`**.

## Находки

### C1 — Critical — `claim` блокирует строку заказа, но проверяет «водитель занят» на незаблокированном запросе ✔
`car_orders/services/overlay.py:37-62` и его двойник из воркера `car_orders/dispatch.py:95-110`.
Оба делают `select_for_update()` на **целевой строке заказа** (`order_id=…`), затем валидируют
правило «один активный заказ» **отдельным, незаблокированным** запросом
(`OrderMeta.objects.filter(driver_id=…).exclude(terminal)`). Два конкурентных claim'а,
назначающих *два разных* заказа *одному* водителю, каждый блокирует свою строку, оба
читают «не занят», оба коммитятся → водитель заканчивает с двумя активными заказами. Никакого
ограничения на уровне БД, поддерживающего правило, нет.
**Эксплойт:** double-tap / два устройства / гонка воркер-против-диспетчера → водитель забронирован дважды.
**Набросок исправления:** обеспечить инвариант на уровне БД (частичное unique-ограничение на `driver_id WHERE
trip_state NOT IN terminal`, с обработкой IntegrityError), либо заблокировать весь активный набор
водителя (например, `select_for_update` по строкам водителя) внутри одной транзакции.
*Не воспроизводится на SQLite — нужны Postgres + многопоточные клиенты.*

### C2 — Critical (зависимое) — блокировки строк являются no-op на дефолтной БД ✔
`config/settings.py:117-122`. С SQLite каждый `select_for_update()` в
`overlay.claim` / `dispatch.claim` молча игнорируется, так что отсутствует даже *однострочная*
защита. В сочетании с C1 конкурентность claim'а фактически ничем не защищена.
**Действие:** подтвердить, что прод выставляет `DATABASE_URL=postgres://…`; если какое-либо окружение
работает на SQLite, повысить до Critical. Явно указать прод-БД в runbook.

### C3 — Critical — неаутентифицированная запись live-локации / меты любого заказа (IDOR-запись) ✔
`car_orders/views.py:219-261` (`LiveLocationView`, `authentication_classes=[]`,
`AllowAny`) и `OrderMetaView.post` `car_orders/views.py:284-290`. `LiveLocationView`
остаётся открытым **даже когда `REQUIRE_OVERLAY_AUTH=True`** (и `test_auth_bridge.py:130-138`
закрепляет это как задуманное поведение). Любой анонимный вызывающий может сделать `POST {lat,lng,geometry}` для **любого**
id заказа — сместить маркер чужого заказа, внедрить произвольный `geometry` JSON, который
рассылается всем наблюдателям и сохраняется. `OrderMetaView.post` аналогично делает upsert оверлейных
полей для любого id, ограждённый лишь `OverlayAuthenticated` (открыто по умолчанию).
**Эксплойт:** перехват/подмена live-трекинга; фальсификация позиции водителя для прохождения геозоны (см. H4).
**Набросок исправления:** требовать аутентификацию + владение (водитель владеет заказом, либо диспетчер) на путях
записи; если симулятору нужен открытый путь, ограничить его общим секретом / non-prod.

### H1 — High — `trip_state.advance()` не имеет ни блокировки, ни транзакции ✔
`car_orders/services/trip_state.py:165-182`: читает `meta`, выполняет `validate()` (которая делает
проверку-затем-установку «один движущийся рейс» на `:135-148`), затем `update_or_create`. Никакого
`select_for_update`, никакого `atomic()`. Прямой конечный автомат и guard «один движущийся рейс»
— чистые гонки read-then-write: double-tap или два устройства могут оба продвинуть состояние, либо
водитель может войти в два движущихся рейса.

### H2 — High — неатомарная двойная запись proxy+local → split-brain demo/overlay ✔
`car_orders/views.py:114-141` (`admin_approve_overlay` / `reject_overlay`). Вызов шлюза
к demo и локальная мутация оверлея последовательны и неатомарны, без
компенсации. Если `services.overlay.release()` (`:140`) или `OrderMeta.update_or_create`
(`:125`) выбросит исключение после того, как demo уже вернул 2xx, demo и оверлей расходятся навсегда —
например, demo отклонил, но в оверлее по-прежнему `dispatchable=True` → продолжает авто-назначение, ровно
тот баг, который этот hook призван предотвратить. **Не протестировано:** `test_critical_fixes.py` мокает
шлюз статическим 200, так что ветка частичного отказа никогда не выполняется.

### H3 — High — гонка создания `DriverShift` выбрасывает необработанный `IntegrityError` (500) ✔
`car_orders/views.py:1172-1189`. Частичные unique-ограничения
`one_active_shift_per_driver` / `one_active_shift_per_car`
(`car_orders/models.py:326-337`) предварительно проверяются через `.filter(...).exists()`, затем
`DriverShift.objects.create(...)` внутри `atomic()`, **без обработчика `IntegrityError`**.
Конкурентные старты смен (или двойное бронирование машины, проскользнувшее мимо предпроверки) выбрасывают
необработанный `IntegrityError` → 500 вместо чистого `CAR_BUSY`. У оверлейного двойника
`DriverShiftState` (`models.py:383-398`) **нет** такого ограничения вообще — его единственный
guard — `update_or_create` по `driver_id`.

### H4 — High — геозону можно обойти через запись позиции с AllowAny ✔
`car_orders/services/trip_state.py:65-89` читает «свежий GPS-фикс» из `DriverPosition`,
который пишется **AllowAny**-эндпойнтом `LiveLocationView` / `_apply_driver_location`. Через C3
неаутентифицированный вызывающий может сделать POST позиции на целевые координаты и пройти геозону.
*(Вторично, Medium:* проверка отклонения `geometry.min_dist_km_to_polyline`
`car_orders/geometry.py:77-82` измеряет расстояние **только до вершин** полилинии, не до
сегментов — на разреженной/прореженной линии водитель посреди сегмента может считаться сошедшим с маршрута,
вызывая ложные перестроения маршрута.)*

### H5 — High — `REQUIRE_OVERLAY_AUTH=False` является прод-дефолтом ✔
`config/settings.py:197`, `car_orders/permissions.py:15-29`. При дефолте каждый
оверлейный эндпойнт доверяет `driver_id` из тела/query (`acting_driver_id` fallback,
`permissions.py:56-62`), а `MyOverlayOrdersView` (`views.py:792-832`) трактует всех как
диспетчера → полное перечисление активной доски + межводительские чтения/записи. Ничто не предупреждает и не
отказывает, если флаг ушёл в прод выключенным. **Набросок исправления:** дефолтить в `True`, либо жёстко падать при старте, когда
`DEBUG=False`, а флаг выключен.

## Сквозные темы
1. **Область блокировки строки против межстрочных инвариантов** (C1, H1, H3) — блокировки покрывают одну строку, но правила
   охватывают строки; владеть инвариантом «1 водитель = 1 активный заказ» должна БД, а не код приложения.
2. **Неатомарные двойные записи proxy + local** (H2) — каждый hook шлюза, который также мутирует
   оверлей, может уйти в split-brain.
3. **AllowAny + identity, доверенная из тела** (C3, H4, H5, M2) — положение «открыто по умолчанию» превращает
   несколько «работает в dev» поведений в write-IDOR в проде.
4. **Проглатывание исключений, скрывающее сбои под видом безобидных состояний** (M1, и широкий
   `except Exception` в `push_order_route`).

## Оценка набора тестов
- **Конкурентность здесь структурно непротестируема:** SQLite + `django_db` означает, что гонки C1/C2/H1/H3
  не воспроизводятся; зелёный `test_overlay_one_active.py` и т.п. утверждают лишь
  *последовательный* guard — ложная уверенность в ключевом инварианте. Добавить Postgres CI-линию
  с многопоточными клиентами для этих случаев.
- **Пути, проксируемые через шлюз, никогда не отрабатываются:** `test_critical_fixes.py` мокает шлюз,
  так что ветка частичного отказа H2 и реальная обработка 502/timeout не протестированы.
- **AllowAny IDOR закодирован как задуманное поведение** (`test_auth_bridge.py:130-138`), а не помечен как проблема.
- Новый набор, добавленный в этом проходе, закрывает пробелы в *логике* (ветки ошибок жизненного цикла,
  граничные случаи геозоны, математику планирования, fail-closed прав доступа, ограничения моделей, валидацию
  сериализаторов, нативную смену Р1, обёртки команд) — 156 новых тестов, все зелёные — но не может
  закрыть пробелы в *конкурентности* и *положении аутентификации* выше.

## Приоритизированный план устранения
- **P0:** C1 (+ C2 подтвердить прод-БД), C3, H1, H2 — ядро двойного бронирования + write-IDOR + split-brain.
- **P1:** H3 (IntegrityError→CAR_BUSY), H4, H5 (дефолт аутентификации), M2 (mass-assignment).
- **P2:** M1, M3–M6, L1–L3.
Каждое исправление P0/P1 должно поставляться с регрессионным тестом на Postgres-линии (набор на SQLite не может это доказать).
