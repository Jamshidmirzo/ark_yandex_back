## 1. Зачем это нужно (бизнес-обоснование)

Сегодня заказ служебного транспорта идёт «вручную» — звонки, мессенджеры, устные
договорённости. Нет статуса, истории, ответственного и аудита.

**Модуль заменяет это формальным процессом:**

> запрос → согласование диспетчером → броадкаст свободным водителям →
> водитель принимает и выезжает → завершение поездки

Результат: прозрачность (кто, что, когда, на какой машине), контроль загрузки
автопарка и водителей, полная история и уведомления на каждом шаге.

## 2. Действующие лица (роли)

| Роль | Кто это | Что делает |
|---|---|---|
| **Заказчик** (Car Requester) | Сотрудник с точечным правом | Создаёт заявку на машину, видит только свои |
| **Диспетчер** (Car Admin) | Координатор автотранспорта | Видит все заявки, согласовывает / отклоняет |
| **Водитель** (Driver) | Сотрудник в роли «Водитель» | Принимает заявку, выбирает машину, выполняет и завершает поездку |
| **Гараж-менеджер** (Garage Manager) | Ответственный за автопарк | Ведёт машины и типы, назначает водителей, видит отчёты |
| **Администратор** | — | Полный доступ |

> «Водитель» — это **роль (членство в группе), а не отдельная карточка**. Сделать
> сотрудника водителем = одна кнопка «Сделать водителем» на его HR-карточке.

## 3. Сквозной процесс и статусы

```
   ┌─────────┐  отправить   ┌─────────┐  согласовать  ┌───────────────┐
   │  draft  │ ───────────► │ pending │ ────────────► │ awaiting_driver│
   │ черновик│   (автор)    │ у дисп. │  (диспетчер)  │  броадкаст     │
   └─────────┘              └─────────┘               └───────────────┘
        │                        │                            │
        │ удалить                │ отклонить                  │ водитель «Принять»
        │ (только draft)         ▼                            │ (+ выбор машины)
        ▼                  ┌──────────┐                       ▼
     (удалена)             │ rejected │ ◄── отмена автором  ┌──────────────┐
                           │терминал. │     до in_progress  │ in_progress  │
                           └──────────┘                     │ поездка идёт │
                                                            └──────────────┘
                                                                   │ водитель «Завершить»
                                                                   ▼
                                                            ┌──────────────┐
                                                            │  completed   │
                                                            └──────────────┘
```

| Статус | Смысл | Кто перевёл сюда |
|---|---|---|
| **draft** | Автор оформляет | автор (`create`) |
| **pending** | Отправлено диспетчеру, ждёт согласования | **только автор** |
| **awaiting_driver** | Согласовано, ушло в ленту всем свободным водителям | диспетчер (`approve`) |
| **in_progress** | Конкретный водитель принял → поездка идёт | водитель (`accept`, first-come) |
| **completed** | Поездка завершена | **только назначенный водитель** |
| **rejected** | Терминальный отказ | диспетчер (`reject`) или автор (отмена своей) — до `in_progress` |

**Жёсткие правила (важно для согласования):**
1. Диспетчер **НЕ назначает** водителя — он только согласовывает. Водитель
   назначается в момент принятия; **машина уже выбрана водителем на входе в смену**
   (см. Р1 и §6), поэтому при «Принять» авто отдельно не выбирается.
2. Отправленную заявку (`pending` и далее) **никто не редактирует** — только
   approve/reject. Нужно изменить условия → отклонить и создать новую.
3. После `in_progress` отмены/отклонения **нет** — только `completed`.
4. У водителя **нет кнопки «Отклонить»** — только «Принять».
5. Завершить поездку **за водителя не может никто** (даже админ) — это действие с
   физической машиной.

## 6. Ключевая механика (с учётом Р1 и Р3)

**Шаг входа (Р1).** Водитель открывает раздел → модалка «Выбери машину на смену»
(см. §4.0). Выбирает одну из закреплённых за ним машин (`Vehicle.drivers`), которая
сейчас `active` и свободна. После выбора водитель считается «на смене на этой машине».

**Кому заявка видна в ленте `awaiting_driver`** — одновременно:
1. пользователь в роли «Водитель» и **выбрал машину на смену**;
2. у него сейчас **нет активной поездки** (`in_progress`);
3. **тип заявки = тип выбранной машины** (заявки других типов ему не показываются).

**Принятие:** first-come exclusive — кто первый нажал «Принять», за тем заявка
фиксируется; у остальных она мгновенно пропадает из ленты (real-time WS).
**Машину выбирать не нужно** — берётся машина смены. Транзакцией: `status=in_progress`,
`driver=он`, `vehicle=машина смены`, `started_at=now`.

**Слежение (Р3).** Пока заявка в `in_progress`: мобила водителя шлёт GPS-координаты
по WebSocket → бэк хранит последнюю позицию и ретранслирует → заказчик видит точку
на карте (§4.8), обновление автоматическое. По `completed` поток координат
останавливается.

**«Свободен/занят» — вычисляется автоматически**, ручных тумблеров нет:
- водитель занят ⇔ есть его заявка в `in_progress`;
- машина занята ⇔ `active` + привязана к активной поездке.

## 7. Граничные случаи (edge cases)

| Ситуация | Поведение системы |
|---|---|
| Два водителя жмут «Принять» одновременно | Первый — поездка; второй — ошибка «уже принято», карточка исчезает |
| Нет свободных машин нужного типа | Заявка водителям в ленте не показывается |
| Водитель уже на поездке | Лента доступных скрыта, вход ведёт в «Текущую поездку» |
| Диспетчер хочет поправить отправленную заявку | Нельзя — только reject, автор создаёт новую |
| Кто-то пытается завершить за водителя | Запрет (403): только назначенный водитель |
| Заказчик отзывает заявку после согласования | → `rejected`, диспетчеру приходит уведомление |
| Уволили/деактивировали водителя | Перестаёт получать броадкаст; снятие роли вручную — **open issue** |
| Водитель забыл «Завершить» | **Open issue** — авто-таймаут / запрос диспетчера (TBD) |

## 8. Уведомления

- **Заказчику:** согласована → ушла водителям; принята; завершена; отклонена.
- **Водителю:** новая релевантная заявка в ленте; отмена уже принятой им (резерв).
- **Диспетчеру:** новые заявки на согласование; отмена заказчиком после согласования.

## 10. Реализованные доработки (гибрид demo + overlay)

В демо `ark_yandex` работает как **шлюз + overlay**: вход, водители, гараж и базовые
заказы **проксируются** в demo-бэкенд, а новые фичи отдаются локально из `OrderMeta`
по id демо-заказа. Demo — источник правды для авторизации/данных, проксирование
ломать нельзя. Полное API — `INTEGRATION.md` §7.

- **Этапы поездки (`trip_state`):** `assigned → to_client → at_client → in_trip →
  at_destination → completed` (+ `waiting`, `cancelled`). Статусы в UI зависят от
  зрителя (формулировка для водителя ≠ для заказчика). Эндпоинт `/{id}/trip-state/`,
  изменения летят по WebSocket.
- **Геозона прибытия — 100 м.** Кнопки «Я на месте» / «Прибыли на место»
  открываются только в радиусе 100 м от точки и при свежем GPS (точку часто ставят
  внутрь здания — 100 м = «у входа», а не «на пине»).
- **Туда-обратно — один заказ.** `has_return` + точка возврата `return_lat/lng`
  (по умолчанию = подача) + `returning`. **Времени возврата нет** — съёмка
  заканчивается непредсказуемо, водитель сам жмёт «Выехать обратно»; «Завершить» —
  только после обратной ноги.
- **Заполнение простоя — без жёсткой блокировки.** Время «на месте» считается
  свободным (`driving_end = окончание − время_на_месте`), а пересечение по времени
  при приёме — **предупреждение, а не блок** (`overlay-claim` всё равно проходит).
  Жёсткий запрет один: заказ уже взят **другим** водителем. Опоздание с возвратом
  подсвечивается флагом «Под угрозой» (`at_risk`), а не запретом. При этом «ехать
  в двух местах сразу» по-прежнему нельзя (одна активная движущаяся нога).
- **Диспетчерская** (`/orders/fleet`): live-карта всех машин (маркер + маршрут A→B
  + пробки) и карточки с полными данными (откуда→куда, проект, подача / ориент.
  окончание, время на месте, «Туда-обратно» / «Везёт обратно», риск, GPS). Срочные /
  под угрозой / опаздывающие — звук + тост. Доступ: `car_order:list | approve`.
- **Уведомления — водителю И заказчику** (`/ws/notifications/{user_id}/`): смена
  статуса, назначение/снятие, «пора выезжать».
- **Прочее:** «Принять» скрыта у пользователя без машины; срочные (`is_urgent`)
  наверх + красный + звук; пустое время подачи = «сейчас».
- **Симулятор** (тест без телефона): `auto_simulate --interval 1.5` едет по фазам
  (к подаче / с клиентом / обратная нога), `remind_departures --loop` — напоминания.
  Флаг `--loop` у симулятора **не** использовать (перегоняет завершённую ногу заново).
