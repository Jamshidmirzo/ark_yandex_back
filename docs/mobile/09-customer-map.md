# 09 — Карта заказа у заказчика (деталь → карта)

Как заказчику (автору заявки) с экрана **детали заказа** открыть **карту** и в реальном времени
видеть: **статус** заказа, **своё** местоположение, маркер водителя и **маршрут до точки встречи**
(подачи). Это «зеркало» драйверского пути: водитель шлёт GPS ([07-driver-websocket.md](07-driver-websocket.md)),
а заказчик его **слушает**. Диспетчерский аналог — [08 (en)](en/08-dispatcher.md).

Контракт сокета описан в [04-live-tracking.md](04-live-tracking.md) §4.1 и
[06-websockets.md](06-websockets.md) §2 — здесь только то, что специфично для **экрана заказчика**:
вход с детали, свой маркер, точка встречи и **жизненный цикл connect/disconnect**.

| Среда | HTTP base URL | WebSocket base |
|---|---|---|
| Dev (локально) | `http://127.0.0.1:8000/api/v1` | `ws://127.0.0.1:8000` |
| Dev (по сети) | `http://<IP-хоста>:8000/api/v1` | `ws://<IP-хоста>:8000` |
| Прод | `https://<ваш-домен>/api/v1` | `wss://<ваш-домен>` |

> `geometry` — **всегда** GeoJSON `[lng, lat]`; для карты переворачивай в `[lat, lng]`.
> Сервер владеет навигацией — клиент рисует то, что прислали (см. [04](04-live-tracking.md)).

---

## §1. Для кого и что на экране

Экран смотрит **заказчик/автор** одного заказа. Слои карты:

| Слой | Источник | Когда |
|---|---|---|
| Баннер статуса | `trip_state` из сокета (+ статус заказа из REST) | всегда |
| Маркер **точки встречи** (подача 🟢) | `origin_lat/lng` из `meta` ([03 §3.1](03-scheduling-overlay.md)) | всегда, если есть координаты |
| Маркер назначения (🔴, опц.) | `address_lat/lng` из `meta` | опционально |
| Маркер **водителя** | `lat/lng` из сокета | как только водитель назначен и пошёл стрим |
| **Маршрут** (полилайн) | `geometry` из сокета | приходит на connect и на смене этапа/перестроении |
| Маркер **«я»** | GPS телефона (плагин геолокации) | всегда, с разрешением на локацию |

> Заказчик — **наблюдатель**: свой GPS он на сервер **не шлёт** (uplink GPS — это отдельный
> драйверский сокет, [07](07-driver-websocket.md) / [04 §4.2](04-live-tracking.md)). Свой маркер
> рисуется только локально.

## §2. Вход: список → деталь → карта

### Тап по заказу → карточка с данными (web + mobile)

Тап по заказу в списке открывает **деталь** — просто данные о заказе. Тот же набор полей показывают и
веб, и мобилка (в мобилке это bottom sheet `CarOrderDetail`); отличается только подача. Данные — из
`GET /car-orders/{id}/` ([02-car-orders.md](02-car-orders.md) → объект `CarOrder`):

| Поле на карточке | Из ответа | Заметка |
|---|---|---|
| Проект | `project_name` | заголовок |
| Дата/время подачи | `planned_datetime` | ISO-8601 UTC → локальное время |
| Адрес (назначение) | `address` | текстовый адрес точки назначения |
| Заметка | `note` | может быть пустой |
| Тип машины | `car_type.name` | |
| Статус | `status` (+ эффективный, ниже) | бейдж |
| Водитель | `driver.name` | если назначен |
| Машина | `car.model` + `car.plate_number` | если назначена |
| Создал / создан | `created_by.name` / `created_at` | |

- **Эффективный статус (важно).** У overlay-принятого заказа demo-статус остаётся `awaiting_driver` —
  не показывай его как есть, считай по [03 «Эффективный статус для UI»](03-scheduling-overlay.md):
  `meta.overlay_claimed && trip_state ∉ {completed, cancelled}` → показывай как «в пути», конкретный
  этап бери из `trip_state`.
- **Этап и live-данные — из нашего слоя, не из demo-детали:** `GET /car-orders/{id}/meta/`
  (`trip_state`, координаты точек) и `GET /car-orders/{id}/live-location/`
  ([03](03-scheduling-overlay.md) / [04](04-live-tracking.md)).
- **`404` на детали** — заказ уже не существует на demo (reject / снят). **Не показывай ошибкой:**
  тихо убери заказ из списка / «Моих заказов» и обнови список ([02 «Order detail»](02-car-orders.md)).

> Карточка работает и сама по себе (просто данные), и как вход на карту — кнопкой ниже. Сейчас этот
> экран-карточка как отдельный вид **не реализован ни на вебе, ни в мобилке** (в мобилке есть лишь
> bottom sheet `CarOrderDetail`) — это контракт на сборку.

### Кнопка «Показать на карте» → экран карты

С карточки кнопка «Показать на карте» открывает экран карты и передаёт **`order_id`**.

**Когда показывать кнопку.** Когда есть что отслеживать — заказ в работе (по эффективному статусу выше):
- `meta.overlay_claimed && trip_state ∉ {completed, cancelled}` → показывай карту;
- иначе по статусу demo: карта для `awaiting_driver` и `in_progress`, скрыта для
  `draft` / `pending` / `completed` / `rejected`.

До назначения водителя (нет `driver_id` / нет позиции) карта показывает **точку встречи + твой
маркер**; маркер и маршрут водителя появятся, как только пойдёт стрим.

**Что рисуется на карте — по этапу заказа** (когда карта открыта):

| Статус / этап | Что на карте |
|---|---|
| `awaiting_driver` (водителя ещё нет) | точка подачи 🟢 + назначение 🔴 + плановый маршрут, **без машины** |
| `in_progress` | то же + живой маркер водителя (`lat/lng` из сокета) |
| терминальный (`completed` / `rejected` / `cancelled`) | карта скрыта |

> Не привязывай карту к одному `in_progress`: awaiting-заказ тоже должен показывать точку встречи и
> маршрут — иначе заказчик не видит, куда подъедет машина (это и был баг «заказ не виден на карте»).

**Данные для первого кадра карты:** статус / водитель / машина — из детали выше; координаты точек и
`trip_state` — из `meta` ([03 §3.1](03-scheduling-overlay.md)); дальше — сокет (replay-on-connect
отдаёт позицию + маршрут + этап, см. §6).

## §3. Статус заказа — два слоя

1. **Рабочий статус** (`CarOrderStatus`, из REST): `draft · pending · awaiting_driver · in_progress ·
   completed · rejected`. В мобилке это enum `CarOrderStatus` (`titleDisplay`, цвета `getBgColor` /
   `getTextColor`) — `lib/features/car_orders/domain/enum/car_order_status.dart`. Справочник статусов —
   [05-reference.md](05-reference.md).
2. **Live-этап** (`trip_state`, из сокета/`meta`): `assigned → to_client → at_client → in_trip →
   at_destination → waiting → completed / cancelled`. **Баннер на карте ведёт `trip_state`** — он
   меняется в реальном времени.

Подписи для заказчика уже определены — колонка «Заказчик видит» в [03 §3.6](03-scheduling-overlay.md):

| `trip_state` | Баннер заказчику | Что на карте |
|---|---|---|
| `assigned` | Назначен водитель | водитель ещё может не слать позицию |
| `to_client` | В пути к подаче | **маршрут водитель → точка встречи** |
| `at_client` | На подаче | водитель у точки встречи |
| `in_trip` | В пути к месту | маршрут подача → назначение |
| `at_destination` | Прибыл на место | маршрута нет (припаркован) |
| `waiting` | Пауза — ожидание | маршрута нет |
| `completed` | Завершён | закрыть панель, отключиться (§6) |
| `cancelled` | Отменён | закрыть панель, отключиться (§6) |

> Помимо карты заказчик может получать **тосты** о смене статуса: сервер шлёт их и автору заказа
> (`author_id`) на сокет `ws/notify/{user_id}/` — с готовым текстом (`«Водитель выехал к месту
> подачи»` и т.д.). Это фоновые уведомления, см. [06 §4](06-websockets.md); для самой карты хватает
> `trip_state` из сокета заказа.

## §4. Своё местоположение заказчика

Чтобы понимать, где ты относительно точки встречи:

1. Запроси **разрешение на геолокацию** (`while in use`) при открытии карты.
2. Подпишись на поток позиции телефона (watch / significant-change) и рисуй маркер **«я»**.
3. Центрируй карту так, чтобы влезли **и ты, и точка встречи** (а после назначения — и водитель);
   не пересчитывай камеру на каждый кадр, чтобы не «дёргать» тайлы.

Это чисто клиентская часть: данные **никуда не отправляются**. Если разрешение не дали — карта
работает и без маркера «я» (показывай точку встречи и водителя).

## §5. Маршрут до точки встречи

«Точка встречи с водителем» = точка **подачи** (`origin_*` в `meta`, 🟢 в [03 §3.1](03-scheduling-overlay.md)).

Главный маршрут рисует **сервер** и присылает в поле `geometry` сокета. Что это за нога — зависит от
этапа ([04 §4.1](04-live-tracking.md)):
- `assigned` / `to_client` → **водитель → точка встречи** (это и есть «маршрут до точки встречи»);
- `at_client` / `in_trip` → точка встречи → назначение;
- `waiting` / `at_destination` → маршрута нет (припаркован).

Правила отрисовки (как и у диспетчера, [08 §9 (en)](en/08-dispatcher.md)):
- `geometry` приходит на connect и **на каждой смене этапа / перестроении** (водитель ушёл с линии
  >80 м) — пришла новая → **заменяй** линию; не пришла → держи прежнюю и просто двигай маркер.
- Линия уже «подрезана» к текущей позиции машины и прорежена под лимит кадра — рисуй как есть
  (перевернув в `[lat, lng]`).

**Свой маршрут до точки встречи (опционально).** Если нужно показать, как **тебе** дойти/доехать до
подачи, построй ногу `я → точка встречи` через `POST /car-orders/estimate/` ([03 §3.2](03-scheduling-overlay.md)):

```json
// запрос: origin = моя позиция, dest = точка встречи
{ "origin_lat": 41.305, "origin_lng": 69.201, "dest_lat": 41.311, "dest_lng": 69.240, "service_minutes": 0 }
// ответ
{ "distance_m": 1180, "drive_minutes": 4, "duration_minutes": 4,
  "geometry": [[69.201,41.305], ...], "source": "osrm" }
```

Рисуй эту `geometry` своим цветом, отдельно от маршрута водителя. `estimate` — без авторизации.

## §6. WebSocket: подключение и отключение

Ядро экрана. Заказчик слушает **сокет одного заказа** — `read-only`, ничего не шлёт.

### URL

```
ws://<host>:8000/ws/order/<order_id>/track/
```

Новое имя; старый алиас `ws/car-orders/<id>/location/` ещё маршрутизируется (используй новое).
Языковой префикс `/<lang>/` терпится, но лучше без него.

**Сборка URL в мобилке** — переиспользуй готовый хелпер
`buildArkWebSocketUrl` (`lib/features/chats/data/websocket_service.dart`):

```dart
final url = buildArkWebSocketUrl(
  baseApiUrl: endpoints.host,          // хост без /api/v1
  accessToken: storage.accessToken,    // даст ?token=… (см. ниже)
  path: '/ws/order/$orderId/track/',
);
```

> **Про токен.** Downlink-сокеты (`order`, `fleet`, `notify`) сейчас **принимают подключение без
> токена** — валидирует его только драйверский uplink ([08 §3 (en)](en/08-dispatcher.md)). Сокет
> заказа присоединяется по `order_id` из пути и query не читает, поэтому лишний `?token=…` ему **не
> мешает** и остаётся forward-compatible на случай, когда downlink-авторизацию включат. Можно слать
> токен уже сейчас.

### Connect (при открытии экрана)

Подключайся в момент открытия карты (init экрана / build провайдера). Сразу после `accept` сервер
делает **replay**: первым кадром отдаёт последнее известное состояние, поэтому карта не моргает
пустой:

```json
{ "lat": 41.30050, "lng": 69.20050, "last_seen": "2026-06-16T09:05:12Z",
  "geometry": [[69.20,41.30], ...], "trip_state": "to_client", "returning": false }
```

Если по заказу ещё нет ни позиции, ни `meta` — replay-кадра не будет; рисуй точку встречи из `meta`
и жди живых кадров.

### Сообщения (что приходит дальше)

Формат тот же, что в [04 §4.1](04-live-tracking.md) / [06 §2](06-websockets.md) — применяй те поля,
что пришли в кадре:
- **позиция:** `{ "lat", "lng", "last_seen", "geometry"? }` → двигай маркер водителя; пришла новая
  `geometry` → замени полилайн;
- **смена этапа:** `{ "trip_state", "returning" }` → обнови баннер (§3);
- `trip_state: "completed"` / `"cancelled"` → покажи финальный статус и **отключись** (ниже).

### Disconnect (при закрытии экрана)

Закрывай сокет, когда экран уходит — в `dispose()` экрана/провайдера — кодом нормального закрытия
`WebSocketStatus.normalClosure`. Сервер сам убирает тебя из группы заказа
([tracking.py](../../car_orders/ws/tracking.py) `disconnect → group_discard`). Также отключайся на
терминальном `trip_state` (`completed`/`cancelled`), показав финал.

### Реконнект и «живость»

- Обрыв → реконнект с backoff ~2 с (готовая логика в `WebSocketService`); на каждом connect сервер
  снова делает replay, так что «догонять» пропущенное не нужно.
- `last_seen` старше ~30 с → покажи «Связь потеряна» (водитель офлайн / в туннеле), маркер оставь на
  месте.
- На прод-токенах: при `401/403` обнови токен и переподключись (в `WebSocketService` это уже
  встроено через `TokenRefreshedEvent`).

## §7. Реализация во Flutter (подсказки)

Экрана карты в мобилке пока **нет** — его предстоит сделать. Что переиспользовать, а не писать заново:

- **URL и жизненный цикл сокета** — `buildArkWebSocketUrl` + класс `WebSocketService`
  (connect / reconnect+backoff / ping / dispose, рефреш токена) из
  `lib/features/chats/data/websocket_service.dart`. Нюанс: кадры заказа — это **простой**
  `{lat,lng,…}`, а `WebSocketService` парсит входящее в Odoo-`BaseSocketMessageModel`. Поэтому либо
  обобщи модель сообщения, либо сделай отдельный лёгкий order-track клиент по той же схеме
  жизненного цикла.
- **Провайдер** — заверни в Riverpod `orderLiveMapProvider(orderId)`: `connect()` при подписке,
  `ref.onDispose(() => disconnect())` (зеркало `webSocketServiceProvider`). Деталь и статус бери из
  существующего `carOrderDetailProvider`
  (`presentation/providers/car_order_detail_provider/`).
- **Новые зависимости** — карта-SDK (**Yandex MapKit**, под стать `ark_yandex`) для тайлов, маркеров
  и полилайна + плагин геолокации для маркера «я».

Состояние экрана держит: `tripState`, позицию водителя (`lat/lng`), `geometry` (перевёрнутую в
`[lat, lng]`), `returning`, координаты точки встречи/назначения из `meta`, свою позицию (`myLat/myLng`).

## §8. Пример (скелет)

```dart
// открытие экрана → connect
final url = buildArkWebSocketUrl(
  baseApiUrl: endpoints.host,
  accessToken: storage.accessToken,
  path: '/ws/order/$orderId/track/',
);
final ch = WebSocketChannel.connect(Uri.parse(url));

List<List<double>>? route; // [lat, lng] для карты
double? carLat, carLng; String tripState = 'assigned'; bool returning = false;

ch.stream.listen((raw) {
  final m = jsonDecode(raw) as Map<String, dynamic>;
  if (m['lat'] != null) { carLat = (m['lat'] as num).toDouble(); carLng = (m['lng'] as num).toDouble(); }
  if (m['geometry'] != null) {
    route = (m['geometry'] as List)
        .map<List<double>>((p) => [(p[1] as num).toDouble(), (p[0] as num).toDouble()]) // flip [lng,lat]→[lat,lng]
        .toList();
  }
  if (m['trip_state'] != null) tripState = m['trip_state'] as String;
  if (m['returning'] != null) returning = m['returning'] as bool;
  if (tripState == 'completed' || tripState == 'cancelled') {
    ch.sink.close(WebSocketStatus.normalClosure); // финал → отключиться
  }
  setState(() {});
});

// свой маркер — из плагина геолокации (на сервер НЕ шлём)
geo.getPositionStream().listen((p) => setState(() { myLat = p.latitude; myLng = p.longitude; }));

// закрытие экрана → disconnect
@override
void dispose() {
  ch.sink.close(WebSocketStatus.normalClosure);
  super.dispose();
}
```

> На проде токен лучше уже передавать (`buildArkWebSocketUrl` добавит `?token=`), хотя сокет заказа
> его пока не требует (§6).

## §9. Справочник и чек-лист

**Эндпоинты экрана:**

| Метод · путь | Назначение |
|---|---|
| `GET /car-orders/{id}/` | статус, водитель, машина, адрес ([02](02-car-orders.md)) |
| `GET /car-orders/{id}/meta/` | координаты точек + `trip_state` ([03 §3.1](03-scheduling-overlay.md)) |
| `POST /car-orders/estimate/` | свой маршрут «я → точка встречи» (опц., [03 §3.2](03-scheduling-overlay.md)) |
| `GET /car-orders/{id}/live-location/` | REST-fallback позиции, если без сокета ([04 §4.2](04-live-tracking.md)) |

**Сокеты:**

| Путь | Направление | Назначение |
|---|---|---|
| `ws/order/{id}/track/` | downlink | позиция + маршрут + этап одного заказа (этот экран; [06 §2](06-websockets.md)) |
| `ws/notify/{user_id}/` | downlink | фоновые тосты о смене статуса (опц., [06 §4](06-websockets.md)) |

> Алиас `ws/car-orders/{id}/location/` ещё маршрутизируется — используй новое имя.

**Чек-лист интеграции:**

- [ ] На детали — кнопка «Показать на карте» (показ по эффективному статусу, §2), передаёт `order_id`.
- [ ] При открытии — `connect` на `ws/order/{id}/track/`, первый кадр (replay) → отрисовка без пустой карты.
- [ ] `geometry` переворачивается `[lng,lat] → [lat,lng]`; новая `geometry` заменяет линию, иначе двигаем маркер.
- [ ] Баннер ведёт `trip_state` (колонка «Заказчик видит», §3).
- [ ] Свой маркер из геолокации; на сервер ничего не шлём.
- [ ] Точка встречи — `origin_*` из `meta`; (опц.) свой маршрут до неё через `estimate`.
- [ ] `completed`/`cancelled` → финал и `disconnect`.
- [ ] При закрытии экрана — `disconnect` (`normalClosure`); реконнект ~2 с; `last_seen` > ~30 с → «Связь потеряна».
