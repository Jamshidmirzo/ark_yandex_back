# 06 — WebSockets (полный справочник)

Все вебсокеты обслуживаются **нашим сервером (Channels/ASGI)**, не demo и не через
HTTP-шлюз. База — тот же хост, что и HTTP, но схема `ws://` / `wss://`:

| Среда | WS base |
|---|---|
| Dev (по сети) | `ws://<IP-хоста>:8000` |
| Прод | `wss://<домен>` |

> **Lang-префикс ок.** Если приложение по привычке шлёт `…/ru/ws/...` — маршрутизируется как `…/ws/...`
> (мы зеркалим HTTP-нормализацию). Лучше слать без префикса.
>
> **geometry везде — `[lng, lat]`** (GeoJSON). Для карты переверни в `[lat, lng]`.
>
> **Реконнект:** на разрыв — переподключайся с задержкой ~2 c. На каждом connect сервер присылает
> актуальное состояние (позиция/маршрут или снапшот), так что после реконнекта догонять не надо.

Четыре сокета. Два важны мобилке (4.1, 4.2 ниже = §1, §2):

| Сокет | Направление | Кто | Назначение |
|---|---|---|---|
| `ws/drivers/me/location/` | **uplink + ответ** | водитель | шлёт свой GPS, получает маркер + полилайн |
| `ws/car-orders/{id}/location/` | **downlink** | заказчик / диспетчер | живая позиция + маршрут + этап одного заказа |
| `ws/car-orders/fleet/` | downlink | диспетчер | снапшот + апдейты всех активных заказов |
| `ws/notifications/{user_id}/` | downlink | любой юзер | тосты на смену статуса своих заказов |

## §1. Драйверский сокет — `ws/drivers/me/location/` (отправка GPS)

**Двунаправленный, главный для приложения водителя.** Водитель льёт GPS, сервер на каждый кадр
отвечает, куда поставить маркер и какую линию нарисовать.

### Подключение
```
ws://<host>:8000/ws/drivers/me/location/?token=<demo JWT>
ws://<host>:8000/ws/drivers/me/location/?driver_id=670        # dev-фолбэк
```
- Идентификация: `?token=` (валидируется как REST-токен) **или** `?driver_id=` (dev).
  Можно прислать и первым сообщением: `{"token":"…"}` / `{"driver_id":670}`.
- Сразу после connect сервер шлёт: `{ "ok": true, "driver_id": 670 }`.

### Что слать (каждые ~5–10 c, на каждый фикс геолокации)
```json
{ "lat": 41.30050, "lng": 69.20050 }
```

### Что приходит в ответ на каждый кадр
```json
{
  "order_id": 88,
  "trip_state": "to_client",
  "lat": 41.30050, "lng": 69.20050,
  "geometry": [[69.20,41.30], [69.21,41.305], ...]
}
```
- `lat`/`lng` — **где сейчас маркер** (двигай его по линии).
- `geometry` — **полилайн текущего участка**; приходит **только когда изменился** (новый этап или
  подъездной маршрут пересчитался при сдвиге >200 м). Пришёл — **перерисуй** линию; не пришёл —
  оставь прежнюю и просто двигай маркер.
- `order_id`/`trip_state` — текущий активный заказ и его этап. `order_id:null` — у водителя нет
  активного заказа (просто на смене).
- Нет связи → копи фиксы, дошли на реконнекте.

### Flutter
```dart
final ws = WebSocketChannel.connect(
  Uri.parse('ws://$host:8000/ws/drivers/me/location/?token=$jwt'));
ws.stream.listen((raw) {
  final m = jsonDecode(raw);
  if (m['lat'] != null) moveMarker(m['lat'], m['lng']);        // маркер по линии
  if (m['geometry'] != null) drawRoute(m['geometry']);          // перерисовать маршрут
});
// на каждый фикс:
ws.sink.add(jsonEncode({'lat': pos.latitude, 'lng': pos.longitude}));
```

> Альтернатива без сокета — HTTP `POST /drivers/me/location/` (см. [04 §4.2](04-live-tracking.md)).
> Серверная логика идентична; выбери ОДИН способ.

## §2. Сокет заказа — `ws/car-orders/{order_id}/location/` (смотреть за машиной)

**Только приём.** Заказчик / диспетчер подписывается на КОНКРЕТНЫЙ заказ и видит, как едет машина.
`order_id` берёшь из `GET /drivers/me/overlay-orders/` (или из деталей заказа).

### Подключение
```
ws://<host>:8000/ws/car-orders/88/location/
```
Сразу после connect — последняя известная позиция + маршрут (если есть).

### Входящие сообщения
Позиция (двигать маркер):
```json
{ "lat": 41.30050, "lng": 69.20050, "last_seen": "2026-06-13T09:05:12Z", "geometry": [[69.20,41.30], ...] }
```
- `geometry` приходит на connect и **на каждой смене этапа / пересчёте подъезда** — на новом **замени** линию.

Смена этапа (обновить баннер):
```json
{ "trip_state": "in_trip", "returning": true }
```
- `trip_state: "completed"` / `"cancelled"` — закрой панель трекинга, отключи сокет.

> Это «выход»: водитель льёт в §1, а заказчик слушает §2 — сервер их связывает по заказу.

## §3. Сокет диспетчерской — `ws/car-orders/fleet/`

**Только приём, для веб-диспетчерской** (мобилке обычно не нужен).
```
ws://<host>:8000/ws/car-orders/fleet/
```
- На connect: `{ "type": "snapshot", "orders": [ FleetOrder, ... ] }` — все активные заказы + ждущие.
- Дальше: `{ "type": "update", "order_id": 88, "lat":…, "lng":…, "geometry":…, "trip_state":… }` на
  каждое движение/смену этапа любого заказа.

---

## §4. Сокет уведомлений — `ws/notifications/{user_id}/`

**Только приём.** Водитель и автор заказа подписываются на свой `user_id` и получают тост на смену
статуса своих заказов.
```
ws://<host>:8000/ws/notifications/671/
```
Входящие:
```json
{ "order_id": 88, "trip_state": "to_client", "message": "Водитель выехал к месту подачи" }
```

---

## Прод (важно)
- **WebSocket НЕ проксируется** demo-шлюзом. Сокеты завершаются на нашем Channels-процессе — на
  проде reverse-proxy/ingress должен разрешать `Upgrade` на пути `/ws/` и вести его на Channels
  (Daphne/Uvicorn), отдельно от HTTP.
- Многопроцессный прод требует **Redis** channel layer (`REDIS_URL`), иначе рассылка между воркерами
  не дойдёт.
- Авторизация WS на проде — токеном в query (`?token=`); сейчас на драйверском сокете уже так,
  остальные downlink-сокеты планируется закрыть тем же токеном.
