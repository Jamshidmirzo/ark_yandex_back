# 03 — Оверлей: расписание, приём, этапы поездки

Эти эндпоинты обслуживаются **шлюзом локально** (у demo их нет). Все работают по
**id заказа из demo**. Формат ошибок — `{"error": {code, message, details}}`.

> Длительность везде — целое число **минут**. Время — ISO-8601 UTC.
> `geometry` — массив `[lng, lat]` (GeoJSON); для карт переворачивай в `[lat, lng]`.

## Зачем оверлей (коротко)
demo хранит базовый заказ, но не умеет: маршрут A→B, длительность/окна и **этапы поездки**.
Это всё мы храним в `OrderMeta` по id заказа.

> **Модель (обновлено 2026-06):** действует правило **«1 водитель = 1 активный заказ»**.
> Назначение делает **сервер автоматически** (auto-dispatch): ближайшему свободному водителю
> на смене с нужным типом машины. Водителю заказ **приходит уже назначенным** — бери его из
> «Мои заказы» (§3.8) и веди по этапам (§3.6). Брать **второй** активный заказ нельзя, пока
> текущий не завершён (`overlay-claim` вернёт `400 DRIVER_BUSY` — §3.4). Прежняя схема
> «последовательные заказы одной машиной / заполнение дыр на съёмке» **отменена**.

## 3.1 Объект `OrderMeta`

```json
{
  "order_id": 88,
  "driver_id": 671,
  "car_id": 5,
  "car_label": "Cobalt (01A777AA)",
  "overlay_claimed": true,
  "origin_lat": 41.311, "origin_lng": 69.240,
  "address_lat": 41.351, "address_lng": 69.290,
  "has_return": false,
  "return_lat": null, "return_lng": null,
  "returning": false,
  "estimated_duration": 43,
  "service_time": 30,
  "planned_datetime": "2026-06-11T09:00:00Z",
  "latest_start": null,
  "trip_state": "to_client",
  "planned_end": "2026-06-11T09:43:00Z"
}
```
- `origin_*` — координаты точки **подачи** (откуда 🟢); `address_*` — координаты точки
  **назначения** (куда 🔴). Префикс `address` = назначение. У точки подачи отдельного текстового
  адреса нет — только координаты. Текстовый адрес назначения хранит demo в поле `address` (раздел 02).
- `has_return` / `return_lat` / `return_lng` / `returning` — **туда-обратно** одним заказом (§3.6.1):
  `has_return=true` → после доставки водитель ждёт и едет обратно в `return_*` (если `null` — обратно
  к подаче). `returning=true` — водитель уже **на обратной ноге** (выставляется автоматически).
  Времени возврата нет — съёмка заканчивается непредсказуемо, водитель сам стартует обратный путь.
- `driver_id` — выставляется при ЛЮБОМ приёме (для проверки окон).
- `overlay_claimed` — `true` **только** если заказ принят нашим слоем (`overlay-claim`), а не demo.
  По нему отличай «ведём у себя» от обычного demo-claim.
- `trip_state` — этап поездки (см. §3.6). Терминальные: `completed`, `cancelled`.

### Прочитать / записать
- `GET /car-orders/{id}/meta/` → объект или `null`.
- `POST /car-orders/{id}/meta/` — upsert, шли только нужные поля.

**Когда писать meta:** сразу **после создания заявки** (`POST /car-orders/` вернул `id`) сохрани
координаты точек и длительность — иначе маршрут/трекинг для заказа не построятся:
```json
{ "origin_lat":41.311, "origin_lng":69.240, "address_lat":41.351, "address_lng":69.290,
  "estimated_duration":43, "service_time":30, "planned_datetime":"2026-06-11T09:00:00Z" }
```

## 3.2 Авто-расчёт маршрута и длительности — `estimate`

`POST /car-orders/estimate/` — **без авторизации**.
```json
{ "origin_lat":41.311, "origin_lng":69.240, "dest_lat":41.351, "dest_lng":69.290, "service_minutes":30 }
```
Ответ:
```json
{ "distance_m":8508, "drive_minutes":13, "service_minutes":30, "duration_minutes":43,
  "geometry":[[69.240,41.311], ...], "source":"osrm" }
```
`source`: `osrm` (точный маршрут) или `haversine` (запасной прямой расчёт).

## 3.3 Проверка окна перед приёмом — `claim-check`

`POST /car-orders/{id}/claim-check/` `{ "driver_id": 671 }`
```json
{ "ok": true,  "conflict": null }
{ "ok": false, "conflict": { "order_id":90, "planned_start":"...", "planned_end":"...", "address":"Заказ #90" } }
```
Информационная проверка окна заказа против активных заказов водителя (+ запас на переезд).

> С правилом **«1 водитель = 1 активный заказ»** (см. вверху) это в основном справочная проверка:
> если у водителя уже есть активный заказ, приём второго всё равно будет заблокирован на сервере
> (`overlay-claim` → `400 DRIVER_BUSY`, §3.4) независимо от результата claim-check. Используй её для
> подсказки в списке, но не как разрешение.

**Батч для списка заказов** (чтобы не дёргать по одному):
- `POST /car-orders/claim-check-batch/` `{ "driver_id":671, "order_ids":[88,90] }` →
  `{ "results":[ { "order_id":88, "ok":true, "conflict":null }, ... ] }` — помещается ли каждый в окно.
- `POST /car-orders/meta-batch/` `{ "order_ids":[88,90] }` → `{ "results":[ OrderMeta, ... ] }` —
  оверлей разом для всех (эффективный статус строки списка).

## 3.4 Приём заказа

Чаще всего принимать вручную **не нужно** — сервер сам назначает заказ водителю (auto-dispatch),
и он появляется в «Мои заказы» (§3.8) уже на `trip_state=assigned`. Дальше веди по этапам (§3.6).

Если приём всё-таки нужен (ручной сценарий):

| Случай | Как принимать | Результат |
|---|---|---|
| Машина **свободна** | demo-`claim`: `POST /car-orders/{id}/claim/` `{car_id}` (раздел 02) → потом `POST /meta/ {driver_id}` | demo `in_progress` |
| Назначение нашим слоем | **`overlay-claim`** (ниже) | принят нашим слоем, `overlay_claimed=true`, `trip_state=assigned` |

### `overlay-claim`
`POST /car-orders/{id}/overlay-claim/`
```json
{ "driver_id":671, "car_id":5, "car_label":"Cobalt (01A777AA)" }
```
- `{ "ok": true, "conflict": null, "meta": {...} }` — принят.
- `400 DRIVER_BUSY` — у водителя **уже есть активный заказ** (правило «1 активный»). Сначала заверши
  текущий (`completed`) — тогда можно взять следующий.
- `400 ALREADY_CLAIMED` — заказ уже взят **другим** водителем (и ещё активен).
- Повторный вызов тем же водителем на **тот же** заказ — идемпотентно, текущий этап не откатывается.

---

## 3.5 Снять заказ / вернуть в очередь — `overlay-release`

`POST /car-orders/{id}/overlay-release/` (без тела)
```json
{ "ok": true, "meta": { "overlay_claimed": false, "driver_id": null, "trip_state": "cancelled", ... } }
```
Очищает наш claim: заказ перестаёт занимать расписание и его перестаёт вести симулятор.

**Вызывай его на teardown-действиях:** при demo-`reject`, при «отменить», при «вернуть в очередь»,
а также можно после завершения. Идемпотентно (если meta нет — просто `{ "ok": true }`).

## 3.6 Этапы поездки — `trip-state`

`POST /car-orders/{id}/trip-state/` `{ "trip_state": "to_client" }` → обновлённый `meta`.
Изменение **пушится в реальном времени** по WebSocket (раздел 04).

Подписи **зависят от роли**: водитель видит действие от первого лица, заказчик/наблюдатель —
нейтральный статус. Каждой фазе — свой цвет тега (соседние фазы различимы, пауза выделена).

| trip_state | Водитель видит | Заказчик видит | Цвет тега | Кнопка водителя → следующий |
|---|---|---|---|---|
| `assigned` | Принят | Назначен водитель | default | «Выехал к клиенту» → `to_client` |
| `to_client` | Еду к клиенту | В пути к подаче | geekblue | «Я на месте» → `at_client` |
| `at_client` | Жду клиента | На подаче | cyan | «Начать поездку» → `in_trip` |
| `in_trip` | Везу клиента | В пути к месту | blue | «Прибыли на место» → `at_destination` |
| `at_destination` | На месте | Прибыл на место | lime | обычный: «**Завершить**»; туда-обратно: «**Выехать обратно**» → `in_trip` (см. §3.6.1) |
| `waiting` | На паузе | Пауза — ожидание | orange | (необязательная ручная пауза) «Продолжить» → `in_trip` |
| `completed` | Завершил | Завершён | green | — |
| `cancelled` | Отменён | Отменён | red | — (выставляется `overlay-release`) |

> Обычный заказ (без обратки) на `at_destination` **завершается** — никакого `waiting`/«продолжить».
> `waiting` остался только как ручная пауза.

**Сервер теперь проверяет этапы строго (не только UI):**
- **Порядок этапов** — только по цепочке выше. Прыжок (напр. `to_client → in_trip` мимо `at_client`)
  → `400 INVALID_TRANSITION`. Идемпотентный повтор того же этапа — ок.
- **Завершение** — `completed` только из `at_destination` (а для туда-обратно — после обратной ноги),
  иначе `400 INVALID_TRANSITION`. `400 INVALID_STATUS` — у уже завершённого заказа.
- **Только назначенный водитель** (или диспетчер) двигает этап — иначе `403`.
- **Геозона прибытия (жёсткий блок на сервере)**: `at_client` / `at_destination` принимаются **только**
  когда у водителя **свежий GPS** (≤120 c) в радиусе **100 м** от точки подачи/назначения — иначе
  `400 TOO_FAR` или `400 NO_FRESH_GPS`. Поэтому **обязательно** стримь GPS (§04). В UI пока далеко —
  показывай расстояние вместо кнопки. (Радиус — `CAR_ORDER_ARRIVAL_GEOFENCE_M`, `0` отключает для тестов.)
- Плановое время (**мягко, не блок**): выехать можно и раньше `planned_datetime`. Если выезжаешь
  сильно заранее (более чем за **30 мин** до подачи) — показывается «выезжаете заранее»; а когда
  приехал на подачу **до** планового времени — «подождите ещё ≈ N мин» (до `planned_datetime`).
  Принять/начать всё равно можно — это только уведомление.
- Один активный заказ на водителя: второй взять нельзя, пока текущий не завершён (`overlay-claim`
  → `400 DRIVER_BUSY`, §3.4). «Отвезти и забрать» — это **один заказ туда-обратно** (§3.6.1), а не
  два отдельных.

## 3.6.1 Туда-обратно (один заказ)

`has_return=true` → после доставки на место водитель **ждёт** во время съёмки, потом везёт **обратно**
(в `return_*`, по умолчанию — к точке подачи). **Времени возврата нет** — съёмка кончается
непредсказуемо, водитель сам стартует обратный путь.

Поток этапов:
```
… → in_trip → at_destination ──«Выехать обратно»──▶ in_trip (returning=true) ──«Прибыли»──▶ at_destination ──«Завершить»──▶ completed
```
- На `at_destination` при `has_return && !returning` кнопка — «**Выехать обратно**» (а НЕ «Завершить»).
  Шли `POST /trip-state/ {"trip_state":"in_trip"}` — бэкенд сам выставит `returning=true` и **пушит его
  по WS** (раздел 04), фронт переключается мгновенно.
- На обратной ноге (`returning=true`) `in_trip` ведёт `назначение → точка возврата`; геозона «Прибыли»
  считается от **точки возврата**.
- На `at_destination` уже при `returning=true` (доехал обратно) кнопка — «**Завершить**» (§3.7).
- `returning` сбрасывается в `false` при повторном приёме/`overlay-release`.

## 3.7 Завершение заказа

- Заказ принят через **demo** (`overlay_claimed=false`): `POST /car-orders/{id}/complete/` (demo) **и**
  `POST /trip-state/ {completed}` — чтобы оверлей не разошёлся.
- Заказ принят через **наш слой** (`overlay_claimed=true`): только `POST /trip-state/ {completed}`
  (demo про него не знает).
> demo разрешает `complete` **только назначенному водителю** — админ/диспетчер demo-заказ не завершит.

---

## 3.8 «Мои заказы» — активные заказы водителя

`GET /car-orders/drivers/me/overlay-orders/?driver_id=671` → массив `OrderMeta` водителя, исключая
`completed`/`cancelled`:
```json
[ { "order_id":88, "trip_state":"to_client", "car_label":"Cobalt (01A777AA)",
    "planned_datetime":"...", "planned_end":"...", "overlay_claimed":true }, ... ]
```
Включает **и demo-принятые, и overlay-принятые** заказы (у обоих есть `driver_id`). Используй для
экрана «Мои заказы» — показывай №, этап (`trip_state`), окно времени, машину, ссылку на деталь.

## 3.9 Продлить / Переназначить заказ

`POST /car-orders/{id}/extend/` `{ "minutes": 30 }` → `{ "ok": true, "meta": {...}, "conflict": null }`
Добавляет минуты к `estimated_duration` (сдвигает `planned_end`). Продление применяется **всегда**;
`conflict` (если не `null`) — предупреждение, что новое окончание пересеклось со следующим заказом
водителя. Доступно водителю или диспетчеру. `400 VALIDATION` — `minutes` не положительное или у
заказа нет сохранённого окна.

`POST /car-orders/{id}/reassign/` (без тела) → `{ "ok": true, "meta": {...} }`
Диспетчер снимает заказ с текущего водителя и **возвращает его в очередь** на авто-распределение:
`overlay_claimed=false`, `driver_id=null`, `dispatchable=true`, а `meta.trip_state` **в ответе** =
**`assigned`** (не-терминальный «ожидает») — в отличие от `overlay-release`, который ставит
терминальный `cancelled`. По WS при этом уходит кадр `trip_state: "cancelled"` — он лишь сообщает
текущим наблюдателям, что прежнее сопровождение завершено (водитель снят); сам заказ снова доступен
авто-диспетчеру. Работает для overlay-принятых заказов (demo-claim принадлежит demo и отсюда не
переназначается). `400 NOT_FOUND` — нет meta.

## Эффективный статус для UI (важно)
У overlay-принятого заказа демо-статус остаётся `awaiting_driver`. Не показывай его — считай так:
- если `meta.overlay_claimed && trip_state ∉ {completed, cancelled}` → показывай **как «в пути»**,
  а конкретный этап бери из `trip_state`;
- иначе — показывай статус заказа из demo.
