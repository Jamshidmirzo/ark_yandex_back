# 03 — Расписание и оверлей (наши фичи)

Эти эндпоинты обслуживаются **шлюзом локально** (demo их не имеет). Все они работают по
**id заказа из demo**. Формат ошибок — `{"error": {code, message, details}}`.

> Длительность везде — в **минутах** (целое число). Время — ISO-8601 UTC.

## 3.1 Авто-расчёт маршрута и длительности

`POST /car-orders/estimate/` — **без авторизации** (чистая функция).

Запрос:
```json
{ "origin_lat": 41.311, "origin_lng": 69.240, "dest_lat": 41.351, "dest_lng": 69.290, "service_minutes": 30 }
```
Ответ `200`:
```json
{
  "distance_m": 8508,
  "drive_minutes": 13,
  "service_minutes": 30,
  "duration_minutes": 43,
  "geometry": [[69.240,41.311], [69.241,41.312], ...],
  "source": "osrm"
}
```
- `geometry` — полилиния маршрута в порядке **[lng, lat]** (GeoJSON). Для большинства карт
  (Leaflet/Yandex) переверни в **[lat, lng]**.
- `duration_minutes` = в пути + на месте. Используй для подстановки длительности в форме.
- `source`: `osrm` (точный маршрут) или `haversine` (запасной прямой расчёт, если роутинг недоступен).

## 3.2 Оверлей заказа (координаты, окно, статус) — `meta`

Хранит то, чего нет в demo: точки A→B, плановое окно, текущий этап поездки.

### Прочитать
`GET /car-orders/{id}/meta/` → объект или `null`:
```json
{
  "order_id": 88,
  "driver_id": 671,
  "car_id": 5,
  "car_label": "Cobalt (01A777AA)",
  "origin_lat": 41.311, "origin_lng": 69.240,
  "address_lat": 41.351, "address_lng": 69.290,
  "estimated_duration": 43,
  "service_time": 30,
  "planned_datetime": "2026-06-11T09:00:00Z",
  "latest_start": null,
  "trip_state": "to_client",
  "planned_end": "2026-06-11T09:43:00Z"
}
```

### Записать (upsert, частично)
`POST /car-orders/{id}/meta/` — отправляй только нужные поля:
```json
{
  "origin_lat": 41.311, "origin_lng": 69.240,
  "address_lat": 41.351, "address_lng": 69.290,
  "estimated_duration": 43,
  "service_time": 30,
  "planned_datetime": "2026-06-11T09:00:00Z"
}
```
**Когда вызывать:** сразу **после создания заявки** (`POST /car-orders/` вернул `id`) сохрани сюда
координаты точек с карты и длительность. Иначе маршрут/трекинг для заказа работать не будут
(demo координаты не хранит).

## 3.3 Проверка окна перед приёмом — `claim-check`

`POST /car-orders/{id}/claim-check/`
```json
{ "driver_id": 671 }
```
Ответ:
```json
{ "ok": true, "conflict": null }
```
или при пересечении по времени с другим заказом водителя:
```json
{
  "ok": false,
  "conflict": { "order_id": 90, "planned_start": "...", "planned_end": "...", "address": "Заказ #90" }
}
```
**Логика:** считает окно `[planned_datetime, planned_end]` этого заказа и сверяет с другими
заказами водителя (+ запас на переезд). Позволяет брать **второй заказ**, если он влезает в
свободный промежуток. Вызывай **перед** открытием выбора машины: `ok:false` → покажи конфликт.

## 3.4 Последовательный заказ той же машиной — `overlay-claim`

Если водитель хочет взять заказ B **той же машиной**, не завершив A (demo это запрещает) —
принимай через наш слой:

`POST /car-orders/{id}/overlay-claim/`
```json
{ "driver_id": 671, "car_id": 5, "car_label": "Cobalt (01A777AA)" }
```
Ответ при успехе:
```json
{ "ok": true, "conflict": null, "meta": { /* OrderMeta */ } }
```
при пересечении окон:
```json
{ "ok": false, "conflict": { "order_id": 90, ... } }
```

**Как использовать в модалке «Принять»:**
- Машина **свободна** (`is_available: true`) → обычный demo-`claim` (`/claim/` `{car_id}`).
- Машина **занята** (твоя текущая) → `overlay-claim`. Заказ становится «активным» в нашем слое
  (см. `trip_state`), demo при этом не трогаем.

После `overlay-claim` заказ ведётся по `trip_state` (ниже), а демо-статус заказа остаётся прежним.

## 3.5 Этапы поездки — `trip-state`

`POST /car-orders/{id}/trip-state/`
```json
{ "trip_state": "to_client" }
```
Ответ — обновлённый `meta`. Изменение **пушится в реальном времени** по WebSocket (см. [04](04-live-tracking.md)),
так что заказчик/диспетчер сразу видят этап.

Поток этапов (что показывать водителю и заказчику):

| trip_state | Заказчик видит | Кнопка водителя → след. статус |
|---|---|---|
| `assigned` | «Водитель назначен» | «Выехал к клиенту» → `to_client` |
| `to_client` | «Водитель в пути к вам» | «Я на месте» → `at_client` |
| `at_client` | «Водитель приехал, ожидает» | «Начать поездку» → `in_trip` |
| `in_trip` | «В пути» | «Прибыли на место» → `at_destination` |
| `at_destination` | «Приехали на место» | «На ожидание» → `waiting` |
| `waiting` | «Водитель отъехал — вы на ожидании» | «Продолжить» → `in_trip` |
| `completed` | «Заказ завершён» | — |

Рекомендация: кнопки «Я на месте»/«Прибыли» разблокируй только когда водитель в радиусе
~400 м от точки (геозона — считай по `live-location` и координатам из `meta`).
