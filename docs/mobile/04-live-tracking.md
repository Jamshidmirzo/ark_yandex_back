# 04 — Live-трекинг: позиция водителя (REST + WebSocket)

**Два направления:**
- **Аплинк (телефон → сервер): отправка GPS.** Два равноценных способа — выбери один:
  - **WebSocket** — `ws://<host>/ws/driver/track/` (канонический; алиас `ws/drivers/me/location/`,
    см. §4.2 и [07](07-driver-websocket.md)). Открой сокет на смене и шли `{lat,lng}` кадрами.
  - **HTTP** — `POST /drivers/me/location/` (§4.3). По одной точке; удобно складывать в офлайн-очередь.
  > Оба прогоняют одинаковую серверную логику (позиция + привязка к заказу + рассылка + маршрут).
- **Даунлинк (сервер → диспетчер/заказчик): WebSocket.** Сервер, получив позицию, **раздаёт** её в
  реальном времени тем, кто смотрит карту заказа (§4.1).

«Живую» карту рисует сервер; позиция/маршрут хранятся в нашем слое по id заказа.

## 4.1 WebSocket (рекомендуется)

```
ws://127.0.0.1:8000/ws/car-orders/{order_id}/location/      (dev)
wss://<ваш-домен>/ws/car-orders/{order_id}/location/        (прод)
```

- Подключайся, когда заказ **в пути** (демо `in_progress` или наш `trip_state` != completed/assigned).
- **Сразу после подключения** сервер пришлёт последнюю известную позицию + маршрут.
- Дальше каждое обновление позиции и каждое изменение этапа пушится отдельным сообщением.
- Авторизация для WS сейчас не требуется (dev). На проде добавим токен в query — заложите место под него.

### Формат входящих сообщений (JSON)

Сообщение с позицией:
```json
{ "lat": 41.32219, "lng": 69.20615, "last_seen": "2026-06-11T09:05:12Z", "geometry": [[69.24,41.31], ...] }
```
- `geometry` — маршрут **текущего участка**, который считает и шлёт **сервер** (он владеет
  навигацией). Приходит в первом сообщении после connect **и заново на каждой смене этапа**
  (`trip-state`): подача (`assigned`/`to_client`) → `позиция водителя → точка подачи`; поездка
  (`at_client`/`in_trip`) → `подача → назначение`; обратка → `назначение → точка возврата`. На
  стоянке (`waiting`/финальный `at_destination`) маршрута нет. Между сообщениями **сохраняй**
  последний полученный, рисуй его как линию; при новом `geometry` — **замени**.

Сообщение со сменой этапа:
```json
{ "trip_state": "in_trip", "returning": true }
```
- Приходит при вызове `trip-state/`. Обновляй баннер статуса у заказчика.
- `returning` — флаг **обратной ноги** туда-обратно (раздел 03 §3.6.1). Приходит вместе со сменой
  этапа; `returning:true` после `at_destination` означает, что водитель поехал назад (точка возврата).
  Сохраняй его, как `geometry`.
- `trip_state: "completed"` — заказ завершён; `trip_state: "cancelled"` — заказ снят
  (`overlay-release`). На обоих закрывай панель трекинга и отключай WS.

### Рекомендации по UI
- Маркер водителя двигай **плавно** (интерполяция между точками за ~1.5 c), а карту центрируй
  только когда водитель подходит к краю — иначе перегрузишь тайлы карты.
- «Связь потеряна» — если `last_seen` старше ~30 c.
- Переподключайся при разрыве (reconnect с задержкой 2 c).

### Flutter — пример
```dart
final ch = WebSocketChannel.connect(
  Uri.parse('ws://127.0.0.1:8000/ws/car-orders/$orderId/location/'));
List<List<double>>? geometry;
double? lat, lng; String? tripState;
ch.stream.listen((raw) {
  final m = jsonDecode(raw);
  if (m['lat'] != null) { lat = (m['lat'] as num).toDouble(); lng = (m['lng'] as num).toDouble(); }
  if (m['geometry'] != null) geometry = (m['geometry'] as List).map<List<double>>(
      (p) => [(p[0] as num).toDouble(), (p[1] as num).toDouble()]).toList();
  if (m['trip_state'] != null) tripState = m['trip_state'];
  if (m['returning'] != null) returning = m['returning'] as bool; // обратная нога
  setState(() {});
});
```

## 4.2 Аплинк по WebSocket (отправка GPS)

`ws://<host>:8000/ws/driver/track/` (канонический; `ws/drivers/me/location/` — рабочий алиас) —
**отдельный** сокет для ОТПРАВКИ позиции (не путать с сокетом заказа из §4.1, который только для приёма).

- Идентификация в query: `?token=<demo JWT>` (валидируется как REST-токен) **или** `?driver_id=670`
  (dev-фолбэк). Можно прислать и первым JSON-сообщением: `{"token":"…"}` / `{"driver_id":670}`.
- После connect сервер шлёт `{ "ok": true, "driver_id": 670 }`.
- Дальше шли позицию кадрами: `{ "lat": 41.331, "lng": 69.255 }` (раз в 5–10 c). На каждый кадр в
  ответ — `{ "updated_orders": [88] }`.
- Делает ровно то же, что HTTP-хартбит: пишет `DriverPosition`, цепляет к активному заказу водителя,
  рассылает по WS наблюдателям и (пере)считывает подъездной маршрут «к клиенту».
- Lang-префикс ок: `ws://host/ru/ws/drivers/me/location/` тоже маршрутизируется.

```dart
final up = WebSocketChannel.connect(
  Uri.parse('ws://$host:8000/ws/driver/track/?token=$jwt'));
// на каждый фикс геолокации:
up.sink.add(jsonEncode({'lat': pos.latitude, 'lng': pos.longitude}));
```

> Нет связи → копи фиксы и дошли, когда сокет восстановится (reconnect c задержкой). Альтернатива —
> HTTP ниже (§4.3 REST), логика на сервере идентична; выбери ОДИН способ, не дублируй.

## 4.3 REST (запасной вариант)

### Получить позицию
`GET /car-orders/{id}/live-location/` → `null` или:
```json
{ "lat": 41.351, "lng": 69.290, "last_seen": "2026-06-11T09:43:00Z", "geometry": [[69.24,41.31], ...] }
```
Можно поллить раз в 3 c, если не используешь WebSocket.

### Отправить позицию (приложение водителя) — ОСНОВНОЙ способ
`POST /car-orders/drivers/me/location/` (с `Authorization: Bearer <токен>`)
```json
{ "lat": 41.331, "lng": 69.255 }
```
- Приложение **водителя** шлёт сюда свои GPS периодически (~раз в 5–10 c) — **один эндпоинт, не нужно
  знать id заказа**. Водитель берётся из токена (`driver_id` в теле — только запасной вариант в dev).
- Сервер на каждый хартбит делает **две вещи**:
  1. сохраняет **позицию по водителю** (`DriverPosition`) — её использует Диспетчерская для подбора
     **ближайшего свободного** водителя. Поэтому шли GPS **и когда водитель просто на смене без
     активного заказа** — иначе он не появится в «Рекомендуем».
  2. прицепляет позицию к **активному** заказу водителя (любой не-терминальный этап, не только
     `to_client`/`in_trip`) — `OrderLiveLocation` — и рассылает по WebSocket (даунлинк). С правилом
     «1 водитель = 1 активный заказ» это ровно его текущий заказ, поэтому карта едет на любом этапе.
- Ответ: `{ "updated_orders": [88] }` — к каким заказам применилось (обычно один). Если ничего не
  везёт — `{ "updated_orders": [] }` (но позиция по водителю всё равно сохранена).

**Фон и офлайн (рекомендация мобилке):**
- На смене запусти фоновую геолокацию (нативный background-location / `watchPosition`), частота
  ~5–10 c или по значимому смещению. Сошёл со смены — останови.
- Нет сети (тоннель, мёртвая зона) → **складывай POST'ы в очередь** и досылай, когда связь вернётся.
  Для этого случая HTTP-аплинк удобнее (легко копить в очередь); сокет-аплинк (§4.2) — равноценная альтернатива.

> **Чтение позиций парка (для диспетчерского клиента, не для водителя):**
> `GET /car-orders/drivers/positions/?max_age=180` → `{ "671": {lat,lng,last_seen}, ... }` —
> последняя позиция по каждому водителю (для подбора ближайшего). `max_age` (сек) отсекает старые фиксы.

### Отправить позицию в конкретный заказ (альтернатива)
`POST /car-orders/{id}/live-location/`
```json
{ "lat": 41.331, "lng": 69.255 }
```
- То же самое, но явно по id заказа (используется симулятором). Тоже разошлёт по WebSocket.
- Можно один раз приложить маршрут: `"geometry": [[lng,lat], ...]` (необязательно).

> `geometry` в ответах/маршруте — в порядке **[lng, lat]**. Для отрисовки на карте переверни в [lat, lng].

## Откуда берётся маршрут (geometry)
Если у заказа сохранён `meta` с координатами A→B, маршрут считается автоматически
(см. [03](03-scheduling-overlay.md) → `estimate`). Если координат нет — трекинг покажет только точку
водителя без линии. Поэтому при создании заявки **сохраняй координаты в `meta`**.

## 4.4 Симулятор (для тестов без телефона)
> **Выключен по умолчанию (обновлено 2026-06):** реальные телефоны шлют GPS на
> `/drivers/me/location/`, и симулятор бы с ними конфликтовал. Чтобы запустить его для теста
> **без телефона** — `AUTO_SIMULATE_ENABLED=1` или флаг `--force`. На реальных водителях не запускать.

`python manage.py auto_simulate --force` непрерывно «возит» каждый активный заказ — как будет слать
мобилка. Симулятор **driver-centric и учитывает этап**, у водителя одна машина = одна позиция,
которая **переносится между заказами**:

- этап `to_client` — едет от **текущей позиции водителя** к точке **подачи** (origin). Это же и
  «пустой» перегон **между заказами**: завершил заказ 1 в его точке назначения → едет к подаче
  заказа 2, и это **видно** на карте;
- этап `in_trip` — едет от **подачи** (origin) к **назначению** (address).

Стоящие этапы (`assigned`/`at_client`/`waiting`/`at_destination`) и терминальные
(`completed`/`cancelled`) — стоят на месте. Позиция переживает рестарт симулятора (берётся из
последней сохранённой live-location водителя).

> Для мобилки это прозрачно: телефон просто шлёт **реальные GPS** (раздел 4 выше), а этапы двигает
> водитель кнопками (`trip-state/`). Симулятор лишь имитирует тот же поток, чтобы подключение
> приложения прошло без сюрпризов.
