# 04 — Live-трекинг: позиция водителя (REST + WebSocket)

Позиция водителя и маршрут хранятся в нашем слое по id заказа. Два способа получать:
**WebSocket** (рекомендуется — push в реальном времени) и **REST** (запасной/одноразовый).

## 4.1 WebSocket (рекомендуется)

```
ws://127.0.0.1:8000/ws/car-orders/{order_id}/location/      (dev)
wss://<ваш-домен>/ws/car-orders/{order_id}/location/        (прод)
```

- Подключайся, когда заказ **в пути** (демо `in_progress` или наш `trip_state` != completed/assigned).
- **Сразу после подключения** сервер пришлёт последнюю известную позицию + маршрут.
- Дальше каждое обновление позиции и каждое изменение этапа пушится отдельным сообщением.
- Авторизация для WS сейчас не требуется (dev). На проде добавим токен в query — заложите место под него.

### Формат входящих сообщений (JSON)

Сообщение с позицией:
```json
{ "lat": 41.32219, "lng": 69.20615, "last_seen": "2026-06-11T09:05:12Z", "geometry": [[69.24,41.31], ...] }
```
- `geometry` приходит **один раз** (в первом сообщении после connect) — это маршрут A→B. В
  последующих сообщениях его нет, **сохраняй** ранее полученный.

Сообщение со сменой этапа:
```json
{ "trip_state": "at_client" }
```
- Приходит при вызове `trip-state/`. Обновляй баннер статуса у заказчика.
- `trip_state: "completed"` — заказ завершён; `trip_state: "cancelled"` — заказ снят
  (`overlay-release`). На обоих закрывай панель трекинга и отключай WS.

### Рекомендации по UI
- Маркер водителя двигай **плавно** (интерполяция между точками за ~1.5 c), а карту центрируй
  только когда водитель подходит к краю — иначе перегрузишь тайлы карты.
- «Связь потеряна» — если `last_seen` старше ~30 c.
- Переподключайся при разрыве (reconnect с задержкой 2 c).

### Flutter — пример
```dart
final ch = WebSocketChannel.connect(
  Uri.parse('ws://127.0.0.1:8000/ws/car-orders/$orderId/location/'));
List<List<double>>? geometry;
double? lat, lng; String? tripState;
ch.stream.listen((raw) {
  final m = jsonDecode(raw);
  if (m['lat'] != null) { lat = (m['lat'] as num).toDouble(); lng = (m['lng'] as num).toDouble(); }
  if (m['geometry'] != null) geometry = (m['geometry'] as List).map<List<double>>(
      (p) => [(p[0] as num).toDouble(), (p[1] as num).toDouble()]).toList();
  if (m['trip_state'] != null) tripState = m['trip_state'];
  setState(() {});
});
```

## 4.2 REST (запасной вариант)

### Получить позицию
`GET /car-orders/{id}/live-location/` → `null` или:
```json
{ "lat": 41.351, "lng": 69.290, "last_seen": "2026-06-11T09:43:00Z", "geometry": [[69.24,41.31], ...] }
```
Можно поллить раз в 3 c, если не используешь WebSocket.

### Отправить позицию (приложение водителя)
`POST /car-orders/{id}/live-location/`
```json
{ "lat": 41.331, "lng": 69.255 }
```
- Приложение **водителя** шлёт сюда свои GPS-координаты (например, раз в 10 c, пока заказ активен).
- Этот POST **сам разошлёт** позицию всем подключённым по WebSocket — ничего дополнительно слать не нужно.
- Можно один раз приложить маршрут: добавь поле `"geometry": [[lng,lat], ...]` (необязательно).

> `geometry` в ответах/маршруте — в порядке **[lng, lat]**. Для отрисовки на карте переверни в [lat, lng].

## Откуда берётся маршрут (geometry)
Если у заказа сохранён `meta` с координатами A→B, маршрут считается автоматически
(см. [03](03-scheduling-overlay.md) → `estimate`). Если координат нет — трекинг покажет только точку
водителя без линии. Поэтому при создании заявки **сохраняй координаты в `meta`**.
